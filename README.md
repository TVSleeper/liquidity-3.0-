# NES/USDT PancakeSwap Infinity Liquidity Console

Локальное приложение для управления концентрированной ликвидностью PancakeSwap Infinity v4 на BNB Smart Chain.

## Запуск

```bash
npm install
npm run compile:contracts
npm run dev
```

Откройте адрес, который покажет Vite, обычно `http://127.0.0.1:5173`.

## Что умеет

- Подключение MetaMask в BNB Smart Chain.
- Восстановление `PoolKey` по `poolId` через `Initialize`-лог `CLPoolManager`.
- Добавление ликвидности в минимальном допустимом tick-диапазоне.
- Добавление двумя токенами или одним токеном через диапазон выше/ниже текущей цены.
- Частичное и полное снятие ликвидности.
- Развертывание персонального `AtomicLiquidityExecutor`.
- Атомарный сценарий `снять ликвидность -> продать выбранный токен -> вернуть остатки`.
- Монитор swap-событий и подготовка атомарного rebalance-транзакта.
- Follow-price range service: автоматическая перестановка выбранной NFT-позиции на заданный процент ниже/выше текущей цены.
- Autonomous keeper bot: отдельный локальный бот без MetaMask, который вызывает rebalance через strategy contract.

## Найденные параметры NES/USDT pool

- `poolId`: `0x1c0195a12979e395d956a9e2581ef720b925b2a7cc1c60a5b82d9fc0fc564ffa`
- `currency0 / NES`: `0x3131f6B80C26936aB03F7d9D29Eb4Ddf36AC3FB5`
- `currency1 / USDT`: `0x55d398326f99059fF775485246999027B3197955`
- `hooks`: `0xb0BAa371b899950B4Ef6A27c21bAf5ef7c434d0f`
- `fee`: `67`
- `parameters`: `0x00000000000000000000000000000000000000000000000000000000000a0045`
- `tickSpacing`: `10`

Один обычный tick — это примерно `0.01%`, но этот pool разрешает usable ticks с шагом `10`, поэтому минимальный фактический диапазон позиции в нём около `0.100045%`.

## Важное ограничение

Локальный интерфейс с MetaMask не может гарантированно выполнить действие в том же блоке сразу после чужого swap: MetaMask требует ручную подпись, а порядок транзакций контролирует сеть/валидаторы. Для настоящего автоматического same-block исполнения нужен отдельный signer-бот с приватным ключом, private orderflow/bundle-инфраструктура или hook, подключенный к пулу при его создании. Для уже существующего пула hook нельзя добавить задним числом.

`AtomicLiquidityExecutor` делает действия атомарными внутри вашей собственной транзакции: если какая-то часть не проходит, откатывается всё.

## Follow-price range

Новый блок `Follow-price range` держит выбранную позицию на заданном смещении от текущей цены:

- `Ниже цены`: позиция ставится ниже текущего tick и использует `token1`/USDT. Если после снятия есть NES, helper продаёт NES в USDT перед новым mint.
- `Выше цены`: позиция ставится выше текущего tick и использует `token0`/NES. Если после снятия есть USDT, helper покупает NES перед новым mint.
- `Offset, %`: насколько далеко от текущей цены ставить минимальный диапазон.
- `Check, sec`: как часто локальный сервис проверяет цену.
- `Mint safety, %`: запас при расчёте новой liquidity, чтобы транзакция реже падала из-за округления/движения цены.

Для работы нужен развернутый `AtomicLiquidityExecutor` и `Approve NFT` для текущей позиции. При каждой автоматической перестановке MetaMask всё равно попросит подпись.

## Автономный keeper-бот

Полностью автономный режим не использует MetaMask для каждой перестановки. Вместо этого:

- `AutonomousRangeStrategy` хранит LP-NFT и остатки NES/USDT.
- `owner` — ваш основной кошелёк. Только он может поставить паузу и вывести NFT/токены.
- `keeper` — отдельный hot wallet бота. Он может только запускать `rebalance` в разрешённом `poolId`.
- `AtomicLiquidityExecutor` выполняет атомарную связку `снять -> swap -> mint`.

Подготовка:

```bash
npm run compile:contracts
cp bot/env.example .env
```

Заполните `.env`:

- `BOT_PRIVATE_KEY`: приватный ключ отдельного bot wallet.
- `OWNER_ADDRESS`: ваш основной кошелёк, который сможет вывести средства.
- `SIDE`: `below` для USDT ниже цены, `above` для NES выше цены, `auto` для автоматического направления по движению цены.
- `AUTO_INITIAL_SIDE`: стартовая сторона для `SIDE=auto`, пока бот ещё не увидел движение цены.
- `OFFSET_PERCENT`: отступ от текущей цены.
- `REBALANCE_ON_UNWANTED_ASSET=true`: бот сработает не только при смещении диапазона, но и когда в позиции появился “не тот” токен. Для `below` он продаёт NES в USDT; для `above` покупает NES за USDT.
- `REMOVE_MIN_BPS=0`: снятие ликвидности не блокируется из-за того, что цена уже успела изменить состав позиции. Защиту цены для swap всё равно задаёт `SLIPPAGE_BPS`.
- `MINT_MAX_BUFFER_BPS`: небольшой запас для округления PancakeSwap при повторном mint, по умолчанию `5`.
- `DRY_RUN=true` сначала оставьте включённым.
- `RECOVERY_SCAN_LIMIT`: сколько последних NFT бот проверяет, если strategy указывает на старую пустую NFT.

Для вашей текущей схемы с USDT ниже цены обычно нужен режим:

```bash
SIDE=below
CHECK_SECONDS=1
REBALANCE_ON_UNWANTED_ASSET=true
MINT_SAFETY_PERCENT=98
```

Если хотите, чтобы бот сам переключался между USDT ниже цены при падении и NES выше цены при росте:

```bash
SIDE=auto
AUTO_INITIAL_SIDE=below
CHECK_SECONDS=1
```

После каждого rebalance лишние NES/USDT могут временно оставаться на strategy-контракте из-за округлений и slippage. Обновленная strategy перед следующей перестановкой автоматически передает эти остатки executor-у, а бот учитывает их в новом mint, чтобы не оставлять капитал без дела.

Деплой strategy:

```bash
npm run bot:deploy
```

Скрипт выведет `EXECUTOR_ADDRESS` и `STRATEGY_ADDRESS`; добавьте их в `.env`.
После этого с owner-кошелька переведите LP NFT на `STRATEGY_ADDRESS`.
Если обновляете старую установку и нужен новый executor, запустите деплой с `FORCE_NEW_EXECUTOR=true`.

Проверочный запуск без транзакций:

```bash
DRY_RUN=true RUN_ONCE=true npm run bot:run
```

Автономный запуск:

```bash
DRY_RUN=false npm run bot:run
```

Перед реальными суммами проверьте на маленькой позиции. Keeper wallet должен иметь BNB на gas, но не должен хранить ваши основные средства.

Если уже развернутая strategy после первого rebalance пишет `Position #... has zero liquidity`, откройте блок `Управление диапазоном`, нажмите `Найти active NFT`, затем `Сохранить tokenId` с owner-кошелька. Новые контракты после пересборки сохраняют новый tokenId автоматически.

## Если позиции не отображаются

- Нажмите `Найти позиции` или `Полный поиск по Transfer-логам` в блоке `Позиции`.
- Проверьте `Scan from block`: он должен быть меньше блока, где была создана NFT-позиция. Для этого пула по умолчанию используется `48000000`.
- Если знаете NFT `tokenId`, вставьте его вручную в поле `NFT tokenId` и нажмите `Добавить tokenId`.
- Позиции с нулевой ликвидностью не показываются.

## Безопасность

- Не используйте `min out = 0` для реальных сумм.
- Проверяйте адреса `Permit2`, `CLPositionManager`, `UniversalRouter` и `CLPoolManager` перед большими транзакциями.
- Сначала тестируйте маленькой суммой.
- Helper-контракт не хранит средства после успешной операции: он sweeping-ит оба токена обратно владельцу позиции, но всё равно требует NFT approve для конкретной позиции.

## Если пул не загружается

Публичные BNB RPC часто ограничивают диапазон `eth_getLogs`. Приложение читает логи чанками, но стартовый блок всё равно влияет на скорость. Если `PoolKey` не найден, уменьшите `Scan from block`; если поиск идёт слишком долго, увеличьте его ближе к блоку создания пула.
